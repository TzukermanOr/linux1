למה לינוקס, ולמה למתחיל בסייבר

לינוקס היא מערכת ההפעלה הנפוצה ביותר בעולם - אבל לא בהכרח על שולחן העבודה. היא מריצה את הרוב המוחלט של שרתי האינטרנט, את כל 500 מחשבי־העל המהירים בעולם, את רוב תשתיות הענן, ואת ליבת אנדרואיד שעל המכשירים הניידים. עבור עולם הסייבר זו כמעט תנאי סף: רוב היעדים שתתקלו בהם בבדיקות חדירות הם מערכות לינוקס, וכלי ההתקפה והבדיקה (Kali, Parrot, ‏ Metasploit, nmap) רצים על לינוקס.

השליטה ב־שורת הפקודה אינה "נחמדה שיהיה" - היא שפת האם של עבודת התקיפה וההגנה. אוטומציה, ניתוח לוגים, ניצול חולשות, תנועה רוחבית - הכל קורה בטרמינל.

01

מבוא ללינוקס והקרנל

⏱ ~45 דקות

מהי בכלל "לינוקס"? התשובה מורכבת יותר ממה שנדמה - ומכילה בלבול נפוץ בין הקרנל למערכת ההפעלה המלאה. נתחיל מהיסודות ההיסטוריים והארכיטקטוניים.

מהי לינוקס

לינוקס היא משפחה של מערכות הפעלה מבוססות UNIX. כלומר, היא יורשת את העקרונות, המבנה ופילוסופיית העיצוב של UNIX - מערכת ההפעלה שפותחה במעבדות Bell בשנות ה־70 - אבל אינה מכילה ולו שורת קוד אחת מתוך UNIX המקורי. לינוקס נכתבה מאפס כמימוש חופשי בהשראת UNIX, ולכן מתארים אותה כ־Unix-like.

◆ הגדרה: מערכת הפעלה (OS)

שכבת התוכנה שמתווכת בין החומרה (מעבד, זיכרון, דיסק, רשת) לבין התוכנות שאתם מריצים. היא מנהלת משאבים, מאפשרת לכמה תוכנות לרוץ יחד מבלי שידרסו זו את זו, ומספקת ממשק אחיד לגישה לחומרה.

! דיוק עובדתי - מתי נולדה לינוקס

המצגת המקורית כותבת שלינוקס נוצרה ב"אמצע שנות ה־90". זו אי־דיוק. לינוס טורבלדס, אז סטודנט בן 21 מפינלנד, פרסם את ההודעה המפורסמת ב־25 באוגוסט 1991, וגרסת הקרנל 0.01 שוחררה בספטמבר 1991. גרסה 1.0 היציבה יצאה ב־1994. כלומר נקודת הלידה היא 1991, לא אמצע העשור.

היום קיימות עשרות רבות של הפצות (Distributions) שכולן מכילות את ליבת לינוקס (הקרנל), ורובן המוחלט חופשיות וחינמיות. מערכות מבוססות לינוקס מריצות מגוון עצום של מוצרים: טלפונים חכמים (אנדרואיד), מחשבים אישיים, שרתים, מצלמות אבטחה, רכבים, נתבים, טלוויזיות חכמות ומכשירי IoT.

הקרנל - ליבת המערכת

כשאומרים "לינוקס" במובן הצר מתכוונים ל־קרנל (Kernel) - והוא בלבד מה שטורבלדס כתב. הקרנל הוא הליבה של מערכת ההפעלה: הרכיב שרץ ברמת ההרשאה הגבוהה ביותר של המעבד, ומתווך בין כל התוכנות לבין החומרה.

◆ הגדרה: קרנל

הקרנל הוא התוכנה המרכזית שנטענת ראשונה בעת האתחול ונשארת בזיכרון כל הזמן. הוא היחיד שניגש לחומרה ישירות, וכל תוכנה אחרת חייבת "לבקש" ממנו שירותים. ניתן להוריד את קוד המקור שלו מ־kernel.org.

לקרנל ארבע אחריות־ליבה שכדאי לזכור:

ניהול התקנים Device

תקשורת עם חומרה דרך מנהלי התקנים (drivers): דיסקים, כרטיסי רשת, מקלדת, GPU.

ניהול זיכרון Memory

הקצאה ושחרור של RAM לכל תהליך, זיכרון וירטואלי, והפרדה כך שתהליך אחד לא יקרא את הזיכרון של אחר.

ניהול תהליכים Process

תזמון (scheduling) של זמן המעבד בין התהליכים, יצירה וסיום של תהליכים.

קריאות מערכת Syscalls

הממשק שדרכו תוכנות מבקשות שירות מהקרנל (לקרוא קובץ, לפתוח חיבור רשת וכו').

◆ הגדרה: User Space מול Kernel Space

המעבד מפריד בין שתי רמות הרשאה. Kernel Space הוא המרחב שבו רץ הקרנל עם גישה מלאה לחומרה. User Space הוא המרחב המוגבל שבו רצות כל התוכנות הרגילות. כשתוכנה ב־User Space צריכה משהו מהחומרה, היא מבצעת system call וה"מעבר" לקרנל מבוקר ומאובטח. ההפרדה הזו היא מה שמונע מתוכנה אחת לקרוס את כל המערכת.

⚠ זווית סייבר - למה הקרנל הוא הפרס הגדול

קוד שרץ ב־Kernel Space רץ עם הרשאות מלאות. לכן פגיעות בקרנל (kernel exploit) היא בין החולשות החמורות ביותר: ניצול מוצלח לרוב מקנה הרשאות root מיידיות (Privilege Escalation מלא). דוגמאות מפורסמות: Dirty COW (CVE-2016-5195) ו־Dirty Pipe (CVE-2022-0847).

מסקנה מעשית: גרסת הקרנל היא מהדבר הראשון שבודק חדירות מזהה על מכונה (uname -r) - כי גרסה ישנה ולא מעודכנת = מועמדת מיידית לניצול ידוע.

Terminal - זיהוי גרסת הקרנל
kali@kali:~$ uname -a Linux kali 6.6.9-amd64 #1 SMP PREEMPT_DYNAMIC x86_64 GNU/Linux kali@kali:~$ uname -r # רק גרסת הקרנל 6.6.9-amd64 kali@kali:~$ cat /proc/version Linux version 6.6.9-amd64 (gcc 13.2.0) ...

למה כל ההפצות שונות - ובכל זאת אותו דבר

זו אבחנה מרכזית: כל הפצות לינוקס חולקות את אותו גרעין בסיסי - הקרנל. מה שמשתנה ביניהן הוא אוסף החבילות (Packages) שמוסיפים סביב הקרנל: איזה ממשק גרפי, איזה מנהל חבילות, אילו כלים מותקנים מראש, ומדיניות העדכונים. בדיוק כמו שמותגי רכב שונים יכולים להשתמש באותו מנוע אבל עם מרכב, פנים ואבזור שונים לגמרי.

✎ בדיקת הבנה - מודול 1
1. מה הקשר בין "לינוקס" ל־UNIX?
לינוקס היא Unix-like: היא מאמצת את הפילוסופיה והממשקים של UNIX אך נכתבה כמימוש חופשי ועצמאי.
2. תהליך משתמש רגיל רוצה לכתוב לדיסק. מה קורה?
כל גישה לחומרה מתווכת על־ידי הקרנל דרך קריאות מערכת - זו ההפרדה בין User Space ל-Kernel Space.
02

הפצות לינוקס

⏱ ~50 דקות

אם הקרנל הוא המנוע - ההפצה היא הרכב המלא. נכיר את המשפחות הגדולות, נבין מה מבדיל ביניהן, ולמה לבודק חדירות זיהוי ההפצה הוא צעד מודיעיני ראשון.

מהי הפצה (Distribution / Distro)

◆ הגדרה: הפצה

חבילה מלאה ומוכנה לשימוש שמורכבת מ: (1) הקרנל, (2) אוסף תוכנות וכלים מערכת (לרוב מפרויקט GNU), (3) מנהל חבילות, (4) לרוב גם סביבת עבודה גרפית. ההפצה היא מה ש"מתקינים" בפועל - לא הקרנל לבדו.

בגלל ש־GNU סיפק את רוב כלי המערכת סביב הקרנל, יש שמתעקשים לקרוא למערכת המלאה GNU/Linux. זו אינה קפדנות סרק: הקרנל לבדו אינו מערכת שימושית.

המשפחות הגדולות

חשוב להבין שהפצות מתחלקות ל"שושלות". שתי המשפחות העיקריות הן Debian (וכל מה שמבוסס עליה, כולל אובונטו ו-Kali) ו־Red Hat (RHEL ונגזרותיה). ההבדל המעשי הראשון שתפגשו הוא מנהל החבילות.

! דיוק חשוב - מה קרה ל־CentOS

המצגת (משנת 2022) מתארת את CentOS כ"הגרסה הנפוצה ביותר לאחסון שרתי web". המידע הזה מיושן. בסוף 2020 הכריזה Red Hat שהיא מפסיקה את CentOS Linux המסורתי (שהיה שכפול 1:1 של RHEL אחרי שחרורו) ועוברת ל־CentOS Stream - שזורם לפני RHEL ומשמש כעין "גרסת בטא". CentOS Linux 8 הגיע לסוף חיים בדצמבר 2021.

כתוצאה מכך נולדו יורשות שכן שומרות על תאימות 1:1 ל-RHEL: Rocky Linux ו־AlmaLinux. אם תיתקלו היום בשרת "כמו RHEL בחינם", רוב הסיכויים שזו אחת מהן, לא CentOS.

הרכיבים הבסיסיים של הפצה

למרות השוני, רוב מערכות הלינוקס המודרניות מורכבות מאותם ארבעה רכיבי־ליבה. נדגים על אובונטו:

סביבת עבודה DE

ה"שולחן עבודה" הגרפי. באובונטו: GNOME. אחרים: KDE, XFCE, Cinnamon.

סייר קבצים File Mgr

הגרפיקה לעיון בקבצים. באובונטו: Nautilus (Files).

מנהל חבילות Pkg Mgr

התקנה/הסרה/עדכון תוכנה. ב-Debian/Ubuntu: APT. ב-RHEL: DNF/YUM.

אמולטור טרמינל Terminal

החלון שמריץ את ה-shell. באובונטו: GNOME Terminal.

◆ הגדרה: מנהל חבילות

תוכנה שמטפלת בהתקנה, עדכון והסרה של תוכנות - כולל פתרון אוטומטי של תלויות (חבילה X דורשת חבילה Y). זה ההבדל הגדול בין משפחות: apt install בעולם Debian מול dnf install בעולם Red Hat. זכרו את ההבחנה הזו - היא הראשונה שתבלבל מתחילים.

סביבות ייעודיות - Kali Linux

היכולת להתאים אישית את לינוקס מגיעה הרבה מעבר לעיצוב שולחן העבודה. Kali Linux היא הפצה מבוססת Debian שמגיעה עם מאות כלי בדיקת חדירות מותקנים מראש, מסודרים בתפריט לפי שלבי תקיפה (איסוף מידע, חולשות, אפליקציות web, סיסמאות, רשת אלחוטית, exploitation, פורנזיקה ועוד). מבחינות אחרות - מבנה הקבצים, הפקודות, מנהל החבילות - Kali זהה כמעט לחלוטין לאובונטו ולשאר הפצות ה-Debian.

✓ טיפ - Kali היא כלי עבודה, לא "מערכת לחיי היומיום"

Kali מתוכננת לבדיקות חדירות בסביבה מבוקרת. לא מומלץ להשתמש בה כמערכת ההפעלה היומיומית: היא מותאמת לעבודה כ-root, כוללת כלים מסוכנים, ומקבלת עדכונים תכופים מאוד (rolling release). אלטרנטיבה התקפית מוכרת נוספת: Parrot OS.

⚠ זווית סייבר - Distro / OS Fingerprinting

זיהוי ההפצה והגרסה הוא צעד מודיעיני קריטי, כי הוא קובע אילו חולשות רלוונטיות, איזה מנהל חבילות יש, ואיפה קבצים נמצאים. מקורות נפוצים לזיהוי על מכונה שכבר השגתם אליה גישה:

Terminal - fingerprinting מקומי
root@target:~# cat /etc/os-release NAME="Ubuntu" VERSION="22.04.3 LTS (Jammy Jellyfish)" ID=ubuntu ID_LIKE=debian ... root@target:~# cat /etc/issue # באנר התחברות root@target:~# hostnamectl # מידע מסכם root@target:~# lsb_release -a

מרחוק (ללא גישה), זיהוי ההפצה נעשה בעקיפין - למשל מבאנרים של שירותים (גרסת OpenSSH אופיינית להפצה), כותרות HTTP, ובאמצעות nmap -O לזיהוי טביעת אצבע של מערכת ההפעלה.

03

רישוי וקוד פתוח

⏱ ~30 דקות

"חינמי" ו"קוד פתוח" אינם בדיוק אותו דבר, ולא כל הרישיונות הפתוחים מתנהגים אותו דבר. הבנת הרישוי משפיעה ישירות על מה שמותר לעשות עם קוד - ועל יכולת הביקורת האבטחתית שלו.

קוד פתוח וחופשי

מערכות לינוקס מסווגות כמערכות קוד פתוח (open-source) וחופשיות (free). הרישיונות הללו קובעים שהקוד חופשי - כולל הזכות לעיין בו, לשנותו ולהפיץ אותו - וכך גם כל תוצר של הקוד. עם זאת, קיימים רישיונות שונים עם תנאים שונים מאוד זה מזה.

! דיוק - "Free" כמו חופש, לא כמו חינם

המילה האנגלית free דו־משמעית. ב"תוכנה חופשית" הכוונה לחירות (free as in freedom) - חופש להשתמש, ללמוד, לשנות ולהפיץ - ולא בהכרח ל"בחינם" (free as in beer). תוכנה חופשית יכולה להימכר בכסף; מה שחייב להישאר חופשי הוא הקוד והזכויות. סיסמת תנועת התוכנה החופשית: "free as in freedom".

◆ הגדרה: Copyleft

Copyleft הוא עיקרון רישוי שלפיו כל מי שמשנה ומפיץ תוכנה חייב להפיץ את הגרסה החדשה תחת אותו רישיון. כך החירות "נדבקת" וממשיכה הלאה. רישיון copyleft חזק (כמו GPL) מונע מחברה לקחת קוד פתוח, לשנותו ולסגור אותו כמוצר קנייני.

! דיוק - RHEL "מסחרית" אבל קוד פתוח

נקודה שמבלבלת: RHEL היא מוצר מסחרי - אבל הקוד עצמו פתוח. מה שעולה כסף הוא התמיכה והתחזוקה (מנוי), לא הקוד. בדיוק בגלל שהקוד פתוח היו אפשריות נגזרות חינמיות כמו CentOS בעבר ו-Rocky/Alma היום.

⚠ זווית סייבר - קוד פתוח וביקורת אבטחה

קוד פתוח הוא חרב פיפיות אבטחתית. היתרון: כל אחד יכול לעיין בקוד, לבדוק אותו ולגלות חולשות - "many eyes". הסיכון: גם התוקף יכול לעיין באותו קוד בדיוק ולחפש חולשות, ולעקוב אחר ה-commits כדי לזהות תיקוני אבטחה ולתקוף מערכות שעדיין לא עודכנו (n-day exploitation).

הנחה שקוד פתוח = בטוח יותר אינה מובנת מאליה. אסון Heartbleed (2014) ב-OpenSSL הוכיח שגם ספרייה קריטית, פתוחה ובשימוש עצום, יכולה לשאת חולשה חמורה שנים מבלי שאף "עין" תבחין בה.

מערכת הקבצים

⏱ ~55 דקות

בלינוקס "הכל הוא קובץ" - ולכן הבנת מבנה הקבצים שווה זהב. נצעד בעץ התיקיות, נבין מה יושב בכל ענף, ונסמן את הקבצים שתוקף רץ אליהם ראשונים.

מבנה היררכי ושורש אחד

מבנה מערכת הקבצים כמעט זהה בכל הפצות לינוקס, והוא מוגדר בתקן רשמי בשם FHS (Filesystem Hierarchy Standard). המבנה היררכי - עץ הפוך. בשונה מ-Windows, שבו לכל כונן אות משלו (C:\, ‏D:\), בלינוקס יש שורש יחיד שמסומן בלוכסן בודד: / (נקרא root או "השורש"). הכל מסתעף ממנו, וגם דיסקים נוספים "נתלים" (mount) לתוך נקודה בעץ הזה ולא מקבלים אות נפרדת.

◆ הגדרה: "הכל הוא קובץ" (Everything is a file)

עיקרון יסוד של UNIX/לינוקס. לא רק מסמכים הם קבצים - אלא גם התקני חומרה (דיסק = /dev/sda), תהליכים רצים (/proc/1234), מידע מהקרנל, ואפילו חיבורי רשת. רוב הפעולות במערכת מיוצגות על־ידי קובץ אחד או כמה, היושבים בתיקיות מוגדרות. זה מה שמאפשר לטפל בכל דבר בעזרת אותן פקודות פשוטות (קריאה/כתיבה לקובץ).

למרות שהשמות יכולים להשתנות מעט בין הפצות, מספר תיקיות־ליבה תמיד קיימות. הטבלה מרחיבה את זו שבמצגת:

ברגע שמשיגים גישה (אפילו כמשתמש מוגבל), אלו ה"חשודים המיידיים":

  • /etc/passwd - רשימת המשתמשים. קריא לכולם כברירת מחדל (היסטורית הכיל גם סיסמאות מוצפנות).
  • /etc/shadow - גיבובי הסיסמאות (hashes). אמור להיות קריא ל-root בלבד. אם הצלחתם לקרוא אותו - זו חולשת הרשאות חמורה, ואפשר לנסות לפצח את הגיבובים אופליין.
  • ~/.ssh/ - מפתחות SSH פרטיים (id_rsa) = התחברות ללא סיסמה למכונות נוספות.
  • ~/.bash_history - היסטוריית פקודות; לעיתים מכילה סיסמאות שהוקלדו בטעות בשורת הפקודה.
  • /var/log/ - לוגים. הגנתית: ראיות. התקפית: יעד למחיקה/טשטוש עקבות.
  • /etc/crontab ו-/etc/cron.* - משימות מתוזמנות; אם משימה שרצה כ-root מפעילה סקריפט שאתם יכולים לכתוב אליו → הסלמת הרשאות.
✓ טיפ - /proc הוא חלון לתוך המערכת

/proc אינו "באמת" על הדיסק - הוא נוצר בזמן אמת על־ידי הקרנל. /proc/cpuinfo מראה את המעבד, /proc/meminfo את הזיכרון, ו-/proc/[PID]/cmdline את הפקודה המלאה שהריצה תהליך - כולל סיסמאות שהועברו כארגומנט, וזו סיבה טובה לא להעביר סודות בשורת הפקודה.

קישורים: Soft Links ו-Hard Links

מערכת הקבצים מאפשרת ליצור "כינוי" לקובץ קיים. יש שני סוגים שונים לחלוטין, ולמרות שהם נראים דומים בפועל הם פועלים בצורה שונה מתחת למכסה. כדי להבין אותם צריך מושג מקדים אחד.

◆ הגדרה: Inode

בלינוקס שם הקובץ ותוכן הקובץ הם שני דברים נפרדים. כל קובץ אמיתי מיוצג ע"י inode - מבנה נתונים בדיסק שמחזיק את ההרשאות, הבעלות, הגדלים, ומצביעים לבלוקי המידע בפועל. שם הקובץ הוא רק "תווית" שמצביעה ל-inode. ההבחנה הזו היא המפתח להבנת קישורים.

Hard Link: שני שמות, אותו inode

קישור קשיח הוא שם שני (או שלישי, או רביעי) שמצביע על אותו inode. שני הקבצים אינם "העתקים" - הם פשוט שני שמות לאותו תוכן בדיסק. שינוי דרך אחד הוא שינוי גם בשני. מחיקת אחד מהם אינה משפיעה על השני; התוכן באמת נמחק רק כשנמחק השם האחרון. מגבלות: עובד רק על אותה מערכת קבצים (אותו mount), ולא ניתן ליצור hard link לתיקייה.

Symbolic Link (Soft Link): מצביע לפי שם

קישור סמלי הוא קובץ קטן ונפרד שמכיל בתוכו את הנתיב לקובץ היעד. אם היעד נמחק - הקישור "נשבר" (broken symlink) ומפנה לכלום. הוא יכול להצביע גם לתיקיות וגם לקבצים בכוננים אחרים, מה שקישור קשיח אינו יכול.

Terminal - יצירת קישורים והשוואת inodes
root@kali:~# echo "original" > file.txt root@kali:~# ln file.txt hardlink.txt # hard link - בלי דגלים root@kali:~# ln -s file.txt softlink.txt # soft link - דגל -s root@kali:~# ls -li # -i מציג inode 123456 -rw-r--r-- 2 root root 9 file.txt 123456 -rw-r--r-- 2 root root 9 hardlink.txt # אותו inode! 789012 lrwxrwxrwx 1 root root 8 softlink.txt -> file.txt

שתי הוכחות חבויות בפלט: ה-inode (123456) זהה אצל המקור וה-hard link, וסוג הקובץ של ה-soft link מסומן ב-l בתחילת ההרשאות (לעומת - בקובץ רגיל).

⚠ זווית סייבר - קישורים כווקטור התקפה

קישורים סמליים הם מקור קלאסי לפגיעויות symlink race / symlink attack: סקריפט שרץ עם הרשאות root פותח קובץ ב-/tmp, ובינתיים תוקף החליף את הקובץ בקישור ל-/etc/shadow - כעת ה-root כותב למקום הלא נכון. בנוסף, בודקי חדירות מחפשים קישורים סמליים בתיקיות שירות (לדוגמה /var/www) שעלולים לחשוף קבצים מחוץ ל-webroot. הגנה מקובלת היא הפעלת protected_symlinks ב-sysctl ושימוש בקריאת מערכת openat עם הדגל O_NOFOLLOW.

⏱ ~75 דקות

מודל ההרשאות של לינוקס הוא לב ההגנה של המערכת - ולכן גם המטרה המרכזית של הסלמת הרשאות. נתחיל מסוגי המשתמשים, נצלול להרשאות הקבצים, ונגיע ל-SUID - אחד הנושאים החשובים ביותר ל-Privilege Escalation.

! דיוק חשוב לפני שנתחיל - שלושה דברים שונים שנשמעים אותו דבר

בלינוקס המילה "root" מופיעה בשלושה הקשרים שונים, וזו אחת הטעויות הנפוצות ביותר אצל מתחילים. שווה לעצור עליה לפני שנמשיך:

מונחמה זההיכן תפגשו אותו

סוגי משתמשים

לינוקס היא מערכת רב־משתמשתית (multi-user): כמה משתמשים יכולים לעבוד במקביל, והיכולות של כל אחד מוגבלות באמצעות הרשאות. שלושה "שחקנים" עיקריים:

חשבון־על שקיים בכל מערכת לינוקס. מנהל המערכת עם כל ההרשאות. בשורת הפקודה מסומן במוסכמה בסימן #.

$ משתמש רגיל UID≥1000

יכולות מוגבלות לפי הרשאות. בשורת הפקודה מסומן במוסכמה בסימן $.

sudo קבוצה

! דיוק - # ו-$ הם מוסכמה, לא חוק

הסימנים # (root) ו-$ (משתמש רגיל) מגיעים ממשתנה הסביבה PS1 שמגדיר את צורת ה-prompt, ומקורם במוסכמה ותיקה. אפשר לשנות אותם - אז אל תסתמכו רק על הסימן כדי לדעת מי אתם. הדרך הוודאית: הפקודה whoami (מי אני) או id (המזהה וההרשאות שלי).

◆ הגדרה: su מול sudo

su (Switch User) מחליף את זהותכם למשתמש אחר ופותח מעטפת חדשה בשמו (ברירת מחדל: root) - דורש את הסיסמה של היעד. sudo מריץ פקודה בודדת בהרשאות root מבלי לעבור משתמש - ודורש את הסיסמה שלכם. ההבדל האבטחתי מהותי: עם sudo כל פעולה נרשמת ביומן, וההרשאות מדויקות וזמניות.

  7. שם הקובץ/התיקייה.
◆ הגדרה: שלוש שלשות ההרשאה - rwx

תשעת התווים מתחלקים לשלוש קבוצות של 3: בעלים | קבוצה | אחרים. בכל שלשה: r=read (קריאה), w=write (כתיבה), x=execute (הרצה). מקף - במקום אות = ההרשאה כבויה.

דוגמה: -rwxr-xr-- פירושו: הבעלים יכול לקרוא+לכתוב+להריץ; הקבוצה יכולה לקרוא+להריץ; כל השאר יכולים רק לקרוא.

✓ טיפ - הרשאות אוקטליות (המספרים)

כל שלשה מיוצגת גם במספר: r=4, ‏w=2, ‏x=1, מחברים. לכן rwx=7, ‏rw-=6, ‏r-x=5, ‏r--=4. שלוש ספרות = שלוש השלשות. הקובץ -rw-r--r-- הוא 644, וסקריפט -rwxr-xr-x הוא 755. כך משנים: chmod 644 file. את הבעלות משנים עם chown user:group file.

! דיוק - rwx על תיקייה זה לא כמו על קובץ

זו אחת הנקודות שמבלבלות מתחילים יותר מכל. שלוש האותיות בעלות משמעות שונה לחלוטין כשהן על תיקייה:

אותעל קובץעל תיקייה
rקריאת תוכן הקובץהצגת רשימת השמות בתוכה (ls עובד)
wשינוי תוכן הקובץיצירה, מחיקה ושינוי שמות של פריטים בתוך התיקייה
xהרצת הקובץ כתוכנית"כניסה" לתיקייה (cd) וגישה לקובץ ספציפי בתוכה לפי שם

צירופים הגיוניים שכדאי להפנים: r בלי x = רואים את שמות הקבצים אבל אי אפשר להיכנס או לקרוא אותם. x בלי r = אפשר לגשת לקובץ אם יודעים את שמו המדויק, אבל ls לא יעבוד. w בלי x = חסר משמעות מעשית, כי אי אפשר להיכנס כדי ליצור משהו.

טעות פופולרית: מחיקת קובץ דורשת w על התיקייה שמכילה אותו - לא על הקובץ עצמו. לכן אפשר למחוק קובץ קריא־בלבד אם יש הרשאת כתיבה לתיקייה. זה גם הסיבה שתיקיית /tmp סטיקית: כולם יכולים לכתוב בה, אבל הסטיקי ביט (ראו בהמשך) מונע מחיקה של קבצים זרים.

איך לינוקס מריצה תוכנית: executable bit, shebang ו-binaries

כדי שקובץ "ירוץ" בלינוקס - יהפוך לתהליך פעיל - צריך שיתקיימו שני תנאים: (1) שיהיה לו ביט הרצה (x) למשתמש שמנסה להריץ אותו, ו-(2) שהקרנל ידע איך לפענח את התוכן שלו. נפרק את שניהם.

◆ הגדרה: Executable Bit

הביט x בהרשאות הוא ה"מתג" שמסמן לקרנל: "מותר להריץ את הקובץ הזה". בלי הביט הזה הקובץ הוא רק... קובץ - גם אם תוכנו תוכנה מושלמת. מוסיפים את הביט עם chmod +x. דגש חשוב: בלינוקס סיומת הקובץ (.sh, ‏.exe, ‏.py) אינה משנה לצורך הרצה - הקרנל מסתכל על הביט והתוכן, לא על הסיומת.

אחרי שיש ביט הרצה, התוכן עצמו יכול להיות אחד משני סוגים:

1. בינארי מהודר (compiled binary)

קובץ בפורמט ELF (Executable and Linkable Format) - קוד מכונה מהודר שהמעבד מריץ ישירות. כך נראים כמעט כל הכלים ב-/bin וב-/usr/bin. אפשר לבדוק זאת עם הפקודה file.

2. סקריפט פרשני (interpreted script)

קובץ טקסט שמתחיל בשורה מיוחדת בשם shebang: #!/path/to/interpreter. כשהקרנל רואה את שני התווים #! בתחילת קובץ, הוא מבין שזה לא בינארי אלא סקריפט, וטוען את התוכנה שצוינה בשורה (bash, python, perl וכד') כדי שתפענח את שאר הקובץ.

Terminal - מהפעלה לפעולה
root@kali:~# cat script.sh #!/bin/bash # shebang - "תריץ אותי עם bash" echo "Hello from script" root@kali:~# ./script.sh bash: ./script.sh: Permission denied # אין ביט x root@kali:~# chmod +x script.sh # מוסיפים את x root@kali:~# ./script.sh Hello from script root@kali:~# file /bin/ls # מה זה /bin/ls בעצם? /bin/ls: ELF 64-bit LSB pie executable, x86-64, dynamically linked root@kali:~# file script.sh script.sh: Bourne-Again shell script, ASCII text executable
✓ טיפ - למה צריך ./ בהתחלה

בלינוקס, התיקייה הנוכחית אינה נכללת בנתיב חיפוש התוכניות ($PATH) כברירת מחדל - שיקול אבטחה מכוון. לכן הקלדת script.sh סתם תחזיר "command not found", ויש לומר במפורש ./script.sh = "תוכנית בשם script.sh בתיקייה הנוכחית". על PATH לעומק - מודול 6.

⚠ זווית סייבר - shebang כ"פנים" של תוקף

תוקפים מנצלים את ה-shebang כדי להריץ פרשן שאתם לא מצפים לו. דוגמה: סקריפט שנראה תמים כ-bash אבל בעצם מתחיל ב-#!/usr/bin/env python3 וסוחב מנת קוד פייתון מטעה. הצעד הראשון בזיהוי "מה זה בעצם הקובץ הזה?" הוא להציץ בשורה הראשונה (head -n 1) ולהריץ file עליו. בנוסף, סקריפטים עם ביט SUID והרשאות פתוחות הם קרקע פוריה לפגיעויות, כי תוקף יכול לערוך אותם ולהריץ אותם בהרשאות הבעלים.

SUID, SGID ו-Sticky Bit - מעבר ל-rwx

מעבר לתשע ההרשאות הרגילות יש שלושה ביטים מיוחדים. אחד מהם, SUID, הוא מהנושאים החשובים ביותר בכל עולם הסלמת ההרשאות.

◆ הגדרה: SUID (Set User ID)

כשבינארי מסומן ב-SUID, הוא רץ עם הרשאות הבעלים שלו - לא עם הרשאות מי שהריץ אותו. אם הבעלים הוא root, כל משתמש שמריץ אותו מריץ קוד בהרשאות root למשך ריצת התוכנית. בפלט ls -l זה מופיע כ-s במקום ה-x של הבעלים: -rwsr-xr-x.

זה לגיטימי ונחוץ - למשל passwd חייב לכתוב ל-/etc/shadow ששייך ל-root. אבל זו גם חולשת ההסלמה הקלאסית.

⚠ זווית סייבר - הסלמת הרשאות (Privilege Escalation)

זה אחד התרגילים הראשונים שבודק חדירות מבצע אחרי השגת דריסת רגל כמשתמש מוגבל. מחפשים בינארי SUID ששייך ל-root וניתן לנצלו כדי "לברוח" ל-shell של root:

Terminal - חיפוש בינארי SUID
lionk@target:~$ find / -perm -4000 -type f 2>/dev/null /usr/bin/passwd /usr/bin/sudo /usr/bin/find /usr/bin/vim <-- חשוד! vim עם SUID lionk@target:~$ # vim עם SUID של root → ניצול ידוע מ-GTFOBins
  • SUID לא תקין: בינארי כמו find, ‏vim, ‏nmap עם SUID של root = הסלמה כמעט מיידית. הפרויקט GTFOBins מתעד בדיוק איך "בורחים" מכל בינארי כזה ל-shell עם הרשאות.
  • sudo מוגדר רשלנית: תמיד מריצים sudo -l לראות מה מותר לכם להריץ כ-root ללא סיסמה. שורה כמו "(root) NOPASSWD: /usr/bin/vim" = הסלמה (GTFOBins).
  • קבצים world-writable שרצים בהרשאות גבוהות (סקריפט של cron, קובץ שירות) = יעד להזרקת קוד.

SGID (s במקום x של הקבוצה) פועל באותו עיקרון אך עם הקבוצה. Sticky bit (t בסוף, נפוץ על /tmp) מבטיח שרק בעל הקובץ יכול למחוק אותו, גם כשהתיקייה ניתנת לכתיבה ע"י כולם.

✎ בדיקת הבנה - מודול 5
1. מה אומרת ההרשאה -rwxr-x--- במספר אוקטלי?
בעלים rwx=7, קבוצה r-x=5, אחרים ---=0 ⟶ 750.
2. מצאתם בינארי -rwsr-xr-x root root /usr/bin/find. למה זה מעניין תוקף?
find עם SUID של root מאפשר הרצת פקודות כ-root (GTFOBins) - הסלמת הרשאות קלאסית.

ה-CLI, מעטפות והתקנה

⏱ ~50 דקות

הטרמינל הוא הכלי החזק ביותר בלינוקס. נבין מהו, נכיר את המעטפות (shells) השונות, נפענח את מבנה הפקודה, ונסכם את שלבי ההתקנה.

ממשק שורת הפקודה (CLI)

◆ הגדרה: CLI / טרמינל

Command Line Interface - ממשק טקסטואלי למערכת. הטרמינל מקבל פקודות מהמשתמש בצורת טקסט ומחזיר משוב בצורת טקסט. למרות שלרוב מערכות הלינוקס יש ממשק גרפי (GUI), רוב הגדרות המערכת עדיין מתבצעות דרך ה-CLI - והוא חיוני בשרתים שלרוב כלל אין בהם ממשק גרפי.

אפשר להשוות את הטרמינל ל-Command Prompt או PowerShell ב-Windows - אך היכולת שלו רחבה ועוצמתית בהרבה. ברוב משימות התקיפה, ההגנה והאוטומציה, הטרמינל הוא סביבת העבודה היחידה.

◆ הגדרה: מודל השכבות - terminal, shell, bash, zsh

ארבעה מושגים שמתבלבלים זה בזה כל הזמן. ההפרדה ביניהם הופכת את כל ההמשך לפשוט הרבה יותר:

⚠ זווית סייבר - Reverse Shell & rbash

שימו לב ל-rbash ברשימה: זו Restricted Bash - מעטפת מוגבלת שנועדה לכלוא משתמש (אסור cd, אסור להריץ פקודות עם נתיב מלא וכו'). אחת ממשימות התוקף היא "בריחה" מ-rbash. מנגד, כשתוקף משיג shell התקפי על יעד מרוחק (reverse shell), הוא לרוב מקבל מעטפת "מכוערת" וצריך לשדרג אותה ל-shell אינטראקטיבי מלא (TTY) - למשל באמצעות python3 -c 'import pty; pty.spawn("/bin/bash")'.

מבנה הפקודה

כמעט כל פקודה בלינוקס בנויה משלושה חלקים:

מבנה: command [options] [arguments]
kali@kali:~$ ping -c 5 8.8.8.8 └ פקודה └ אפשרות └ ארגומנט
  • פקודה (command): מה לעשות - להציג נתונים, להגדיר, ליצור קובץ, להריץ תוכנית.
  • אפשרויות (options/flags): משנות התנהגות. מיוצגות באות אחת או יותר עם מקף: -c. אפשרות "ארוכה" עם שני מקפים: --count.
  • ארגומנטים (arguments): על מה לפעול - קובץ, כתובת, ערך.
✓ טיפ - קיצורי המקלדת שיחסכו לכם שעות

Tab משלים שמות פקודות וקבצים (לחיצה כפולה = הצגת אפשרויות). ↑/↓ דפדוף בהיסטוריה. Ctrl+R חיפוש בהיסטוריה. Ctrl+C עצירת תהליך. Ctrl+L ניקוי המסך. Ctrl+A/Ctrl+E תחילת/סוף שורה.

איפה לינוקס מחפשת תוכניות: משתנה ה-PATH

נסו לחשוב על שאלה לכאורה תמימה: כשאתם מקלידים ls, איך לינוקס יודעת שהבינארי נמצא בקובץ /usr/bin/ls? למה לא צריך לכתוב את הנתיב המלא בכל פעם? התשובה היא משתנה סביבה בשם PATH - אחד המושגים הבסיסיים ביותר במערכת, ואחד שמבלבל את רוב המתחילים.

◆ הגדרה: PATH

$PATH הוא משתנה סביבה שמכיל רשימת תיקיות, מופרדות בנקודתיים (:). כשמקלידים פקודה ללא נתיב מלא, ה-shell עוברת על התיקיות הללו בסדר, ומפעילה את הבינארי הראשון שנמצא בעל השם המבוקש. אם לא נמצא בשום תיקייה - תופיע ההודעה הידועה command not found.

Terminal - בדיקת PATH ואיתור פקודות
root@kali:~# echo $PATH /usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin root@kali:~# which ls # איפה נמצא הבינארי של ls /usr/bin/ls root@kali:~# type ls # מזהה גם alias וגם built-in ls is aliased to `ls --color=auto` root@kali:~# command -v cd # cd הוא built-in של ה-shell cd

התיקיות הסטנדרטיות שתפגשו ב-PATH בכל מערכת לינוקס:

  • /bin - בינארים בסיסיים, חיוניים גם במצב חירום (ls, cat, cp).
  • /sbin - בינארי ניהול מערכת, לרוב דורשים root (fdisk, iptables).
  • /usr/bin ו-/usr/sbin - רוב הבינארים של המשתמשים והמערכת.
  • /usr/local/bin - תוכנות שהותקנו ידנית, מחוץ למנהל החבילות של ההפצה.
  • בהפצות מודרניות רבות, /bin ו-/sbin הם בעצם symlinks ל-/usr/bin ול-/usr/sbin (איחוד שנקרא usrmerge).

אפשר להוסיף תיקייה ל-PATH לסשן הנוכחי (זמני) או בקובצי האתחול של ה-shell (~/.bashrc, ~/.zshrc) כדי שיישאר קבוע:

Terminal - הוספה ל-PATH
root@kali:~# export PATH=$PATH:/opt/mytools # מצורף בסוף root@kali:~# echo $PATH /usr/local/sbin:/usr/local/bin:/usr/bin:/bin:/opt/mytools
⚠ זווית סייבר - PATH Hijacking ו-Privilege Escalation

הסדר ב-PATH הוא קריטי. ה-shell עוצרת בתיקייה הראשונה שבה היא מוצאת התאמה. דמיינו משימת cron שרצה כ-root עם PATH שבו /tmp מופיעה לפני /usr/bin, ומבצעת tar בלי נתיב מלא. תוקף יצור קובץ זדוני בשם /tmp/tar, יסמן אותו כניתן להרצה - וכשהמשימה תרוץ, היא תפעיל את הגרסה של התוקף במקום את /usr/bin/tar המקורית. זוהי PATH Hijacking - וקטור קלאסי להסלמת הרשאות.

הגנה: בסקריפטים רגישים תמיד להשתמש בנתיב מלא (/usr/bin/tar) או להגדיר במפורש PATH=/usr/bin:/bin בתחילת הסקריפט. בצד התקיפה: בדקו את הסקריפטים שרצים כ-root (sudo -l, ‏cron, שירותי systemd) - אם נמצאת בהם פקודה ללא נתיב, זה כדאי לנסות לנצל.

התקנת לינוקס (Kali) - סקירה

⏱ ~70 דקות

ניווט: pwd ו-cd

השימוש בטרמינל דומה לעיון בסייר הקבצים - אתם תמיד "נמצאים" בתיקייה מסוימת ופועלים ממנה. שתי פקודות הניווט הבסיסיות:

  • pwd (Print Working Directory) - מדפיסה את התיקייה הנוכחית.
  • cd [יעד] (Change Directory) - מנווטת לתיקייה אחרת.
Terminal - ניווט

◆ הגדרה: נתיב מוחלט מול נתיב יחסי

נתיב מוחלט (absolute) מתחיל מהשורש / ומתאר את כל הדרך - למשל /home/lionk/file. הוא תקף מכל מקום. נתיב יחסי (relative) לא מתחיל ב-/ ומתפרש ביחס לתיקייה הנוכחית. הבחנה זו קריטית - טעות נפוצה אצל מתחילים.

וריאציות של cd

הפקודה cd אינה ייחודית ללינוקס (קיימת גם ב-Windows ו-macOS) ומשמעה "change directory". להלן הצורות החשובות:

וריאציהפעולה

✓ טיפ - נקודה אחת ונקודתיים

. (נקודה אחת) = התיקייה הנוכחית. .. (שתיים) = התיקייה שמעל. לכן הרצת סקריפט בתיקייה הנוכחית כותבים ./script.sh - ה-./ אומר במפורש "מכאן", כי לינוקס לא מחפש תוכניות בתיקייה הנוכחית כברירת מחדל (בניגוד ל-Windows) - שיקול אבטחה מכוון.

נתיב מוחלט מול נתיב יחסי

בלינוקס יש שתי דרכים שונות לציין מיקום של קובץ או תיקייה. ההבדל ביניהן נראה קטן אבל הוא קריטי - וזו אחת הסיבות הנפוצות לסקריפטים שלא רצים כשמעבירים אותם בין מערכות.

◆ הגדרה: שני סוגי נתיבים

נתיב מוחלט (absolute path) מתחיל תמיד ב-/ ומפרט את הדרך המלאה מהשורש עד היעד. דוגמה: /home/kali/Documents/notes.txt. הוא עובד מכל מקום ובכל זמן - לא משנה איפה אתם נמצאים בעת ההקלדה.

נתיב יחסי (relative path) אינו מתחיל ב-/, ומתפרש ביחס לתיקייה הנוכחית (pwd). אם אתם ב-/home/kali והקלדתם Documents/notes.txt, לינוקס תפענח את זה כ-/home/kali/Documents/notes.txt. אם תעברו תיקייה - אותה פקודה תתייחס למקום אחר לגמרי.

הצגת תוכן: ls

פקודה חיונית נוספת היא ls שמציגה את תוכן התיקייה. היא בשימוש עם אפשרויות רבות:

⚠ זווית סייבר - הקבצים המוסתרים זה איפה שהסודות

בלינוקס קובץ "מוסתר" הוא פשוט כזה ששמו מתחיל בנקודה (.bashrc, ‏.ssh, ‏.env). אין כאן הסתרה אבטחתית - רק מוסכמה ש-ls רגיל לא מציג אותם. דווקא שם יושבים קבצי ההגדרות והסודות: מפתחות SSH, אסימוני API בקובצי .env, היסטוריית פקודות. לכן הפקודה הראשונה של תוקף בכל תיקיית בית היא ls -la, לא ls.

קבצים נסתרים - מוסכמת הנקודה

נעצור רגע על המושג "קובץ מוסתר", כי זה אחד הדברים הבסיסיים שמתחילים לעיתים מפספסים, ובהמשך נסתמך עליו בלי הסבר.

◆ הגדרה: קובץ מוסתר בלינוקס

בלינוקס "הסתרה" של קובץ היא מוסכמה, לא מנגנון אבטחה: כל קובץ או תיקייה ששמם מתחיל בנקודה (.) פשוט לא מופיעים בתצוגת ברירת המחדל של ls. הקובץ עצמו רגיל לחלוטין, פתוח לקריאה אם יש הרשאות, ויראה בכל ls -a או במנהל קבצים שמופעלת בו אפשרות "הצג מוסתרים".

למה בעצם להסתיר? כדי לא להעמיס. תיקיית הבית של כל משתמש מלאה בעשרות קבצי הגדרות, היסטוריה ומטמון שמתעדכנים כל הזמן. בלעדי המוסכמה הזו ls בתוך תיקיית הבית היה רעש מוחלט במקום רשימה שימושית.

‏mkdir -p /נתיב/עמוק/יותר יוצר את כל תיקיות האב בדרך אם אינן קיימות, במקום ליפול בשגיאה. חיוני ליצירת מבנה תיקיות מקונן בפקודה אחת.

בלינוקס rm מוחק לצמיתות - אין "סל מיחזור". הפקודה rm -rf / (מחיקה רקורסיבית כפויה מהשורש) עלולה למחוק את כל המערכת. הקפידו במיוחד עם הרשאות root ועם תווי wildcard. הגנתית: למחיקה מאובטחת (ללא שחזור) קיים shred.

08

קלט, פלט והפניות

⏱ ~55 דקות

כאן לינוקס הופך מ"שורת פקודה" ל"שפת תכנות". זרמי הנתונים, ההפניות והצינורות הם מה שמאפשר לשרשר פקודות פשוטות לכלי עוצמתי - וזה בדיוק מה שבודק חדירות עושה כל היום.

שלושת זרמי הנתונים

בלינוקס, כל נתון בטרמינל מסווג לאחת משלוש קטגוריות. לכל אחת יש גם מספר - File Descriptor - שחשוב להכיר:

STDIN FD 0

קלט תקני. כל מה שנכנס לתוכנית - למשל מה שמקלידים, או נתונים שמוזרמים אליה. (הארגומנטים בשורת הפקודה אינם נחשבים קלט של הבינארי.)

STDOUT FD 1

פלט תקני. כל מה שמודפס למשתמש ואינו שגיאה. למשל תוצאת ls היא ה-STDOUT שלה.

STDERR FD 2

שגיאה תקנית. סוג פלט מיוחד שמציין תקלה - הרשאה נדחתה, קובץ לא קיים. ניתן לטפל ולסנן אותו בנפרד מהפלט הרגיל.

◆ הגדרה: למה מפרידים פלט משגיאה

ההפרדה בין STDOUT (FD 1) ל-STDERR (FD 2) היא גאונית: היא מאפשרת לשמור את התוצאות לקובץ ובו זמנית לראות שגיאות על המסך, או להפך - לדכא שגיאות מבלי לאבד תוצאות. זה הבסיס לטריק 2>/dev/null שמסתיר שגיאות ("זריקה לפח").

כתיבה וקריאה: echo, cat, >

אפשר להשתמש בעורכי טקסט, אבל לינוקס מספק פקודות פשוטות יותר לטיפול בקלט/פלט:

  • echo "טקסט" - מדפיס טקסט/נתונים ל-STDOUT.
  • cat [קובץ] - לוקח את תוכן הקובץ ומדפיס אותו ל-STDOUT.
  • > [קובץ] - מפנה את הפלט לקובץ במקום למסך.
! תיקון שיבוש במצגת - מה echo באמת עושה

המצגת המקורית כותבת ש-"echo מתארך את הנתונים" (dates the data). זו טעות/שיבוש. ‏echo פשוט מדפיס את הטקסט שקיבל אל ה-STDOUT - שום קשר לתאריך. הפקודה שמדפיסה תאריך היא date. ייתכן שמדובר בבלבול בין echo ל-date בעת כתיבת המצגת.

Terminal - echo, cat, redirection
root@kali:~# echo "Hello Linux" > file_1 # כתיבה לקובץ root@kali:~# cat file_1 Hello Linux root@kali:~# cat file_1 > file_2 # העתקת תוכן root@kali:~# echo "" > file_1 # דריסה -> ריקון הקובץ

הפניית זרמים (Redirection)

ההפניה מתבצעת באמצעות סימנים. שימו לב להבדל הקריטי בין דריסה להוספה:

סימןפעולה
>מפנה פלט (STDOUT) לקובץ - דורס את התוכן הקיים!
>>מפנה פלט לקובץ - מוסיף בסוף (append) בלי לדרוס.
<מפנה קלט (STDIN) - מזין לתוכנית מתוך קובץ.
2>מפנה רק את זרם השגיאות (STDERR).
2>/dev/null"זורק לפח" שגיאות - נפוץ מאוד בכלי תקיפה לשקט במסך.
&>מפנה גם פלט וגם שגיאות יחד לאותו יעד.
✓ טיפ - הסכנה ב-> וההצלה של >>

הבעיה המרכזית עם > היא שהוא דורס מידע. הרצת echo "x" > important.log תמחק את כל הלוג! לכן להוספה משתמשים תמיד ב->>. זוכרים: סימן בודד = דורס, כפול = מוסיף.

צינורות (Pipes)

◆ הגדרה: Pipe (הסימן |)

צינור לוקח את ה-פלט (STDOUT) של פקודה אחת ומזרים אותו כ-קלט (STDIN) של הפקודה הבאה. כך משרשרים פקודות פשוטות לכלי מורכב. לדוגמה: הפלט של ls מוזרם ל-grep כדי לסנן רק שורות מסוימות.

Terminal - pipe
lionk@kali:~$ ls -l | grep file # רק שורות שמכילות "file" -rw-r--r-- 1 lionk lionk 0 Oct 14 file_1 -rw-r--r-- 1 lionk lionk 0 Oct 14 file_2

מניפולציה על זרמים: grep, awk, sed

לינוקס מספק כלים רבי־עוצמה לעיבוד טקסט. שלושת המרכזיים:

grep סינון שורות

הכלי הנפוץ ביותר לסינון. מסנן את הפלט ומציג רק שורות שתואמות תבנית. "מסנן שורות".

awk סינון עמודות

בן־הזוג של grep - מסנן עמודות בתוך הפלט. שימושי במיוחד לסקריפטים, לוגים ונתונים מובנים.

sed עורך זרם

המתקדם מבין השלושה - Stream EDitor. מתאים, מחליף ועורך קלט/פלט תוך כדי זרימתו.

Terminal - דוגמאות שימוש
# grep: כל השורות עם "root" בקובץ המשתמשים lionk@kali:~$ grep root /etc/passwd root:x:0:0:root:/root:/bin/bash # awk: הדפסת העמודה הראשונה בלבד (מופרד ב-:) lionk@kali:~$ awk -F: '{print $1}' /etc/passwd root daemon bin lionk ... # sed: החלפת "old" ב-"new" בכל הפלט lionk@kali:~$ echo "old text" | sed 's/old/new/' new text

חיפוש קבצים: find ו-locate

מכיוון שלינוקס מייצג הכל כקבצים, חיפוש קבצים הוא פעולה נפוצה מאוד.

  • find [מיקום] -name [שם] - חיפוש "חי" בזמן אמת. גמיש ועוצמתי, אך יכול להיות איטי; צריך להשתמש נכון עם אפשרויות ו-wildcards.
  • locate [שם] - מהיר יותר כי הוא שואל מסד נתונים מטמון מוכן מראש. את המסד מעדכנים עם updatedb (ולכן locate לא ימצא קבצים שנוצרו אחרי העדכון האחרון).
Terminal - find ו-locate
root@kali:~# find /home/ -name "file_*" /home/lionk/file_1 /home/lionk/file_2 root@kali:~# updatedb # מעדכן את מסד הנתונים root@kali:~# locate file_ # חיפוש מהיר מהמטמון

שרשור צינורות (Multiple Piping)

אפשר לשרשר כמה צינורות יחד למשימות מורכבות. דוגמה מהמצגת - הפקת רשימת המשתמשים שיש להם קבצים בתיקיית /etc:

Terminal - שרשרת צינורות
root@kali:~# ls -l /etc/ | awk '{print $3}' | awk NF | sort | uniq couchdb dradis postgres root

פירוק השרשרת שלב אחר שלב: ls -l /etc/ מציג את התוכן ⟸ awk '{print $3}' מסנן את העמודה השלישית (הבעלים) ⟸ awk NF מנקה שורות ריקות ⟸ sort ממיין ⟸ uniq מסכם ערכים חוזרים. התוצאה: רשימת המשתמשים הייחודיים שמחזיקים קבצים ב-/etc.

⚠ זווית סייבר - שרשרת הצינורות היא כלי התקיפה היומיומי

שילוב הכלים האלה הוא הלחם והחמאה של עבודת בודק החדירות:

  • ציד סודות: grep -rin "password" /var/www 2>/dev/null - סריקה רקורסיבית של קוד אתר אחר סיסמאות מוטמעות.
  • ציד SUID: find / -perm -4000 -type f 2>/dev/null (מודול 5) - צינור קלאסי להסלמה.
  • ניתוח לוגים: cat access.log | awk '{print $1}' | sort | uniq -c | sort -nr - הפקת כתובות ה-IP הנפוצות ביותר בלוג (הגנתית: זיהוי תוקף; התקפית: זיהוי הגנות).
  • טשטוש עקבות (התקפי, להכרה הגנתית): תוקף עלול להשתמש ב-sed או הפניה כדי לערוך/לרוקן לוגים ב-/var/log ולמחוק את כתובת ה-IP שלו. לכן לוגים קריטיים נשלחים לשרת לוגים מרוחק (SIEM) שהתוקף לא יכול לערוך.
✎ בדיקת הבנה - מודול 8
1. רוצים להוסיף שורה ללוג קיים בלי למחוק אותו. במה משתמשים?
‏> דורס; >> מוסיף בסוף בלי לדרוס.
2. מה עושה 2>/dev/null בסוף פקודה?
‏FD 2 הוא STDERR; הפנייתו ל-/dev/null מסלקת שגיאות מהמסך - נפוץ מאוד ב-find של בדיקות חדירות.

עורכי טקסט ו-Vim

⏱ ~35 דקות

עריכת קבצי הגדרות היא לחם חוקו של כל מי שעובד עם לינוקס. נכיר את העורכים הפשוטים, ואז נצלול ל-Vim - העורך ש"תמיד יהיה שם", גם על השרת המרוחק ביותר.

nano ו-gedit

יש בלינוקס שני סוגי עורכי טקסט - גרפיים ועורכי CLI (בתוך הטרמינל):

  • gedit - עורך גרפי נפוץ, דומה מאוד ל-Notepad. פשוט ואינטואיטיבי, אך דורש סביבה גרפית.
  • nano - עורך CLI ידידותי שמופעל ב-nano [קובץ]. משתמש בקיצורי מקלדת המוצגים בתחתית המסך: Ctrl+O לשמירה, Ctrl+X ליציאה.
◆ הגדרה: Vim - והסיבה שלומדים אותו

‏nano ו-gedit אינם מותקנים מראש בכל ההפצות ולעיתים צריך להתקין אותם ידנית. ‏Vim (או לפחות קודמו Vi), לעומת זאת, קיים כמעט בכל הפצת לינוקס. ‏Vim ו-Vi הם עורכי CLI עתירי תכונות, אך אינם נחשבים ידידותיים למשתמש. השם Vim = "Vi iMproved" (Vi משופר); Vi הוותיק עוד פחות ידידותי.

זו בדיוק הסיבה שכל איש לינוקס חייב לדעת Vim ברמה בסיסית: כשמתחברים לשרת מרוחק מינימלי או לקופסה שפרצתם אליה - לעיתים Vim הוא העורך היחיד שיהיה שם.

שני המצבים של Vim

הדבר שהכי מבלבל מתחילים: Vim עובד בשני מצבים (modes) עיקריים, ואותה הקשת מקלדת עושה דברים שונים לגמרי בכל מצב.

Command Mode ברירת מחדל

המצב שבו Vim נפתח. כאן הקשות נתפסות כפקודות וקיצורים - לא כטקסט. משמש לעריכה מהירה: העתקה, הדבקה, מחיקה, שמירה.

Insert Mode עריכה

המצב שבו מקלידים טקסט רגיל. נכנסים אליו בלחיצה על i מתוך Command Mode, ויוצאים חזרה בלחיצה על Esc.

✓ טיפ - התרשים המנטלי של Vim

תזכרו את הזרימה: נכנסים ל-Vim ⟵ נמצאים ב-Command Mode ⟵ לוחצים i כדי לכתוב (Insert) ⟵ לוחצים Esc כדי לחזור לפקודות ⟵ מקלידים :wq כדי לשמור ולצאת. הטעות הנפוצה ביותר: להקליד בטעות בזמן שאתם ב-Command Mode (במקום ב-Insert), והתווים מתפרשים כפקודות.

פקודות במצב Command

מצב הפקודה רגיש לשני סוגי פקודות:

  • קיצורי מקלדת: לחיצה על תווים מתפרשת כפעולה. למשל dd גוזר שורה (במקום להקליד "dd"), ו-p מדביק טקסט שהועתק (במקום להקליד "p").
  • פקודות נקודתיים: פקודות מיוחדות שמתחילות ב-:, כמו :wq לשמירה ויציאה.

ניווט וחיפוש בטקסט

הסיבה ש-Vim עדיין בשימוש היא הגמישות והעושר שלו. משתמש מנוסה יכול לחפש, להחליף ולקפוץ במהירות בין מיקומים. לקפיצה לשורה מסוימת: :[מספר שורה]. לחיפוש: /[ביטוי] ואז n לתוצאה הבאה.

⚠ זווית סייבר - Vim כמנגנון בריחה

‏Vim אינו רק עורך - הוא מנוע שיכול להריץ פקודות מערכת, וזה הופך אותו לכלי הסלמה:

  • מתוך Vim אפשר להריץ פקודת shell עם :!command (למשל :!id), או לפתוח shell מלא עם :shell.
  • אם משתמש כלוא ב"מעטפת מוגבלת" (rbash) אך מצליח לפתוח קובץ ב-Vim - הוא יכול "לברוח" משם ל-shell חופשי. זו טכניקת shell escape קלאסית.
  • אם sudo -l מראה ש-Vim מורשה להרצה כ-root (או שיש לו SUID) - sudo vim -c ':!/bin/bash' נותן shell של root. הפרויקט GTFOBins מתעד את זה במדויק.

הגנתית: זו הסיבה שאסור לתת sudo על עורכי טקסט, ושיש להקשיח מעטפות מוגבלות.

תרגיל מסכם

✓ משימה - "מהאפס ל-root logic"

נסו לבצע את הרצף הזה במכונת Kali/VM ולהסביר כל שלב:

  1. זהו את ההפצה והקרנל: cat /etc/os-release, ‏uname -r.
  2. בדקו מי אתם ומה מותר לכם: id, ‏sudo -l.
  3. צרו מבנה: mkdir -p lab/logs ואז cd lab.
  4. צרו "לוג" והוסיפו אליו שורות עם > ואז >> - שימו לב להבדל.
  5. סננו את הלוג: cat logs/app.log | grep ERROR | awk '{print $1}' | sort | uniq.
  6. חפשו בינארי SUID: find / -perm -4000 -type f 2>/dev/null.
  7. בדקו תיקיות בית של משתמשים: ls -la /home/* - חפשו .ssh ו-.bash_history.
✎ מבחן סיכום
1. תוקף קרא בהצלחה את /etc/shadow כמשתמש רגיל. מה המשמעות?
‏/etc/shadow אמור להיות קריא ל-root בלבד. גישה אליו = חשיפת גיבובים לפיצוח.
2. מה ההבדל בין su ל-sudo?
su = Switch User (מעטפת חדשה, סיסמת היעד); sudo = פקודה בודדת בהרשאות גבוהות (הסיסמה שלכם, ונרשם ביומן).
3. נתקעתם ב-Vim ואתם לא מצליחים לצאת. מה תקלידו?
קודם Esc כדי לוודא שאתם ב-Command Mode, ואז :q! יוצא בכוח בלי לשמור.
